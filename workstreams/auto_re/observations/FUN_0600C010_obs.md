### Key Finding: Player Physics Runs First

**FUN_0602EEB8 (player physics) is the FIRST game code to execute in the frame.**
It runs for ~7400 instructions before any AI or rendering code. This means:

1. Player physics computes new position/speed/heading BEFORE rendering reads them
2. There's no frame lag between physics and display
3. AI cars (FUN_0600E0C0) update AFTER the player, so they react to the player's
   new state from the same frame
4. Rendering (FUN_0600B6A0) runs last, reading the final computed state

The player physics pipeline is only ~3.4% of the frame — the vast majority of
CPU time is spent in rendering preparation (FUN_0600D336 region, ~202K instructions).
This is consistent with Saturn's VDP requiring CPU-driven polygon submission.

### Time Trial vs Race Mode

In time trial (0 AI cars), the AI iteration loop (FUN_0600E0C0) runs but doesn't
iterate (car count = 1, loop starts at i=1, immediately exits). In race mode with
39 AI cars, FUN_0600E0C0 would consume significantly more CPU time. The player
physics cost is fixed regardless of AI count.

## Other Observations

- FUN_0600C010 is called from 0x06009460 which is in the main game loop.
  The stack shows BIOS → 0x06000310 → 0x060072E4 → 0x0600305C → 0x0600943C →
  FUN_0600C010. This is the standard Saturn game loop chain.
- FUN_0600DF20 appears at idx=7510 AFTER FUN_0600CA96. This is the wrapper
  that calls FUN_0602EEB8 (PR=0x0600E5F8 in the cycle 1 observation matches
  an address within the FUN_0600DF20→FUN_0600E526 chain).
- The AI friction call (FUN_0600CA96) appears between player physics and the
  AI loop, suggesting some shared surface computation runs between the two.
